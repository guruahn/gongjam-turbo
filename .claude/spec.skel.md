# feature 추가를 위한 제안서 스켈레톤

## 개발 진행 규칙

> 아래 규칙을 유념해서 진행해줘.

- "코드를 수정해줘" 라고 지시하기 전까지 수정하지 않는다.
- feature에 대해 약간이라도 이해하지 못한 부분이 있으면 그냥 넘어가지 말고 질문을 통해 명확하게 이해하도록 해줘.
- 큰 틀에서 다음의 순서로 진행한다. 0. feature 이름을 정한다.
  1. 현재 프로젝트에 대한 이해
  2. 추가할 feature에 대한 이해: 아래 "feature정의 템플릿" 항목에 기반해서 대화하여 정리합니다.
  3. feature 추가를 위해 구조 변경이 필요한지 여부
  4. 대략적인 작업 계획을 순서대로 정리해줘.
  5. 작업 단위는 서로 의존성이 없는 선에서 클로드코드가 한 세션에 해결할 수 있을 정도의 사이즈내로 쪼개줘.
  6. feature 추가를 위한 준비가 끝나면 코딩을 시작하기 전에 .claude 폴더에 spec-{FEATURE_NAME}.md 형태로 문서로 남겨줘.
- spec 문서를 작성 완료했다면 세션을 종료해.

## 추가전 참고문서

- 프로젝트 구조: CLAUDE.md
- 프로젝트 분석 리포트
  - .claude/analysis-report.md
  - .claude/codebase-analysis-report.md

## 기능정의 템플릿

### 🎯 feature 정의 관련: 아래 질문을 통해 파악한다

- "이 feature으로 해결하고 싶은 구체적인 문제가 뭐야?"
- "현재 팀에서 수동으로 하고 있는 일 중에 자동화하고 싶은 게 있어?"
- "Slack에서 어떤 알림이 부족하다고 느껴?"

### 🏗️ 기술적 제약 관련: 아래 질문을 통해 파악한다

- "기존 코드의 보안 이슈들을 먼저 해결하고 새 feature을 추가할까?"
- "새 feature이 기존 크론잡 스케줄에 영향을 줘도 괜찮아?"
- "Firebase DB 스키마 변경이 필요하면 어떻게 할까?"

### 📊 우선순위 관련: 아래 질문을 통해 파악한다

- "가장 급한 feature이 뭐야? (팀에서 매일 불편해하는 것)"
- "이 feature이 없으면 팀 업무에 얼마나 지장이 있어?"
- "개발 시간 vs 효과 측면에서 어떤 걸 우선시해?"

### 🔄 운영 관련: 아래 질문을 통해 파악한다

- "새 feature이 잘못 동작하면 팀에 미치는 영향이 어느 정도야?"
- "테스트는 어떻게 할 수 있을까? (테스트 Slack 채널이 있나?)"
- "feature 추가 후 모니터링은 어떻게 할까?"

## 기술적 제한사항

- 함수단위 테스트를 작성한다.
- 엄격한 타이스크립트 적용.
- 기존 feature에 영향이 없어야 한다.
